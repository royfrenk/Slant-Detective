# SD-031 — Provider abstraction layer (LLMProvider interface + factory + storage)

**Priority:** P0 · **Estimate:** 1d · **Sprint:** Week 5 (Multi-provider BYO-key)
**Status:** Ready for Development

---

## TL;DR

Refactor the Anthropic-specific service-worker code into a provider-agnostic `LLMProvider` interface + factory so OpenAI and Gemini can be added as sibling implementations without touching the rubric pipeline. Extend storage to hold per-provider keys + per-provider model selection.

## Current State

- `extension/src/service-worker/anthropic-client.ts` — Anthropic-hardcoded fetch wrapper (URL, headers, `x-api-key`, `anthropic-version`).
- `extension/src/options/validate-api-key.ts` — hardcoded 1-token call to Anthropic.
- `extension/src/service-worker/layer2-pipeline.ts` — calls `callAnthropicMessages` directly.
- `extension/src/service-worker/rubric-prompt.ts` — single prompt (no provider variant hook).
- `extension/src/shared/storage-keys.ts` — single key `anthropicApiKey`.
- `extension/src/service-worker/response-validator.ts` — assumes Anthropic `content[0].text` shape.

## Expected Outcome

- A new `extension/src/service-worker/providers/` module with:
  - `types.ts` — `LLMProvider` interface (`validateKey`, `complete`, `extractText`, `id`, `modelIds`).
  - `anthropic.ts` — refactored Anthropic implementation.
  - `index.ts` — `getProvider(id)` factory.
- Storage schema change: `anthropicApiKey` → `providers: { anthropic: { key, model }, openai?: {...}, gemini?: {...} }` + `activeProvider: 'anthropic' | 'openai' | 'gemini'`.
- Migration: on first load, existing `anthropicApiKey` users get mapped to `providers.anthropic.key` + `activeProvider: 'anthropic'`. Old key deleted after migration confirmed.
- `layer2-pipeline.ts` reads `activeProvider`, calls `getProvider(id).complete(prompt)`, passes response to the same rubric validator.
- Response validator is split: provider-specific text extraction → shared rubric-JSON validator.
- `rubric-prompt.ts` exposes `getRubricPrompt(providerId)` so providers can override (e.g., OpenAI JSON mode hints, Gemini system-instruction placement).
- Zero behavior change for existing Anthropic users (no new UI, no new keys required). All existing tests still pass.

## Integration Points

### LLMProvider interface (to be defined in `providers/types.ts`)

```typescript
export type ProviderId = 'anthropic' | 'openai' | 'gemini'

export interface ProviderCompleteInput {
  system: string
  user: string
  model: string
  maxTokens: number
}

export interface LLMProvider {
  readonly id: ProviderId
  readonly modelIds: readonly string[]          // First entry = default model
  validateKey(key: string): Promise<ApiKeyTestResult>
  complete(input: ProviderCompleteInput, apiKey: string): Promise<unknown>  // Raw provider response
  extractText(response: unknown): string        // Pull the text out of provider-specific shape
}
```

The `complete()` return is `unknown` — caller always calls `extractText()` next, then passes string to `validateRubricResponse()`. This keeps the rubric validator fully provider-agnostic.

### Storage schema (new)

```typescript
// extension/src/shared/storage-keys.ts additions
export const PROVIDERS_KEY = 'providers' as const       // stores ProvidersConfig object
export const ACTIVE_PROVIDER_KEY = 'activeProvider' as const  // stores ProviderId string

// Shape stored at PROVIDERS_KEY:
interface ProvidersConfig {
  anthropic?: { key: string; model: string }
  openai?:    { key: string; model: string }
  gemini?:    { key: string; model: string }
}
```

### Storage migration (runs in `index.ts` `onInstalled` and once at SW startup)

1. Read both `anthropicApiKey` (old) and `providers` (new).
2. If `anthropicApiKey` is set and `providers.anthropic` is not set → write `providers: { anthropic: { key: oldKey, model: 'claude-haiku-4-5-20251001' } }`, `activeProvider: 'anthropic'`, then delete `anthropicApiKey`.
3. If `providers.anthropic` is already set → skip (idempotent).
4. If neither is set → fresh install, do nothing.

The migration must be run-once and idempotent. Best location: a `runStorageMigration()` helper called at the top of `index.ts` (before any message listeners fire) and also in `onInstalled`.

### Cache key extension

Current format: `sd_cache_${urlHash}:${bodyHash}:${rubricVersion}`
New format: `sd_cache_${urlHash}:${bodyHash}:${rubricVersion}:${providerId}:${model}`

`buildCacheKey` in `cache.ts` gains two new parameters. `runLayer2Analysis` in `layer2-pipeline.ts` passes them through from `Layer2Input` (which must be extended with `provider` and `model` fields).

### layer2-pipeline.ts changes

`runLayer2Analysis` signature changes from:
```typescript
runLayer2Analysis(input: Layer2Input, apiKey: string): Promise<RubricResponse>
```
to:
```typescript
runLayer2Analysis(input: Layer2Input, provider: LLMProvider, apiKey: string): Promise<RubricResponse>
```

`Layer2Input` gains `provider: ProviderId` and `model: string` fields (used for cache key). The `fetchAndValidate` inner function calls `provider.complete(...)` then `provider.extractText(...)` then `validateRubricResponse(...)`.

### Error dispatch in `index.ts`

Currently `instanceof AnthropicApiError` is used to classify errors at lines 123-138. After SD-031, the pattern becomes:

- Provider throws its own error class (e.g., `AnthropicApiError`, `OpenAIApiError`, `GeminiApiError`), OR
- Providers can throw a shared `ProviderApiError` with `statusCode` — simpler.

**Recommendation:** Define a single `ProviderApiError extends Error { statusCode: number }` in `providers/types.ts`. All three providers throw this. `index.ts` catches `ProviderApiError` and maps `statusCode` to the existing message reasons (`invalid_api_key`, `rate_limited`, etc.). This avoids `instanceof` for each provider class.

## Implementation Plan

**Progress:** 100%

### Critical Decisions

- **Prompt split:** `system` = all analyst instructions + JSON schema + "Return ONLY a JSON object" suffix (everything above the article block); `user` = `{ARTICLE_TITLE}\n\n{ARTICLE_BODY}` bound to the user role. This split is load-bearing for OpenAI JSON mode (requires "JSON" in system) and Gemini `systemInstruction`. The line boundary inside `rubric-prompt.ts` is between the instruction block that ends with the JSON schema definition and the article-content placeholder that follows.
- **`rubric_v1.0.prompt.txt`:** Delete the stale reference file. Add a one-line comment at the top of `rubric-prompt.ts`: `// Single source of truth for all rubric prompts. See getRubricPrompt(providerId).` Do not create new `.txt` files for SD-033/034 variants.
- **Cache key extension:** `buildCacheKey` gains `provider` and `model` parameters. New format: `sd_cache_${urlHash}:${bodyHash}:${rubricVersion}:${provider}:${model}`. Anthropic entries cached before SD-031 will never match new keys (different segment count) — they expire naturally via `CACHE_MAX_ENTRIES` eviction. Do not force-clear the old cache.
- **`ApiKeyTestResult` widening:** Extend `code` from `401 | 403` to `400 | 401 | 403` in `providers/types.ts`. Gemini returns HTTP 400 for invalid API keys; this must be a typed `invalid` result, not an unclassified error.
- **Error class:** Define a single `ProviderApiError extends Error { statusCode: number }` in `providers/types.ts`. All three providers throw this for HTTP errors. `index.ts` maps `statusCode` to the existing message reasons. `GeminiSafetyError` (SD-034) is a separate class — it is a semantic outcome, not an HTTP error.
- **`getRubricPrompt` return shape:** `{ system: string; user: string; version: string }`. `system` is the fully resolved instruction string (no remaining placeholders). `user` is a template string with `{ARTICLE_TITLE}` and `{ARTICLE_BODY}` placeholders that the caller fills before sending. The `version` string is provider-specific (`'rubric_v1.0'` for Anthropic, `'rubric_v1.0-openai'` for OpenAI, `'rubric_v1.0-gemini'` for Gemini) and is used as `rubricVersion` in `buildCacheKey`.

## Cross-Cutting Concerns for Plan-Writer

1. **`rubric-prompt.ts` return type** — Currently returns `AnthropicRequestBody` (a type from `anthropic-client.ts`). After SD-031, `buildRubricPrompt` must return a provider-agnostic `ProviderCompleteInput`. The old `AnthropicRequestBody` type is no longer the right shape (it includes `messages: AnthropicMessage[]` with Anthropic-specific structure). The Anthropic provider's `complete()` method will internally construct the `AnthropicRequestBody` from a `ProviderCompleteInput`.

2. **`__RUBRIC_VERSION__` and the cache key** — `__RUBRIC_VERSION__` is currently `'v1.0'` (set by `VITE_RUBRIC_VERSION` env var or default). After SD-031, the rubric version for Anthropic stays `'rubric_v1.0'` (or whatever the env var is). SD-033 introduces `'rubric_v1.0-openai'` and SD-034 introduces `'rubric_v1.0-gemini'`. These are provider-specific suffixes returned by `getRubricPrompt(providerId).version`. The `buildCacheKey` call picks up the version from the provider's prompt descriptor, not from `__RUBRIC_VERSION__` directly — so the cache correctly separates outputs. Plan-Writer should decide whether `getRubricPrompt` returns an object `{ system: string, user: string (template), version: string }` or whether each provider embeds its version in a different way.

3. **`api-key-card.tsx` is provider-singleton for SD-031** — Full multi-provider UI is SD-032's job. SD-031 only needs the storage schema change. `api-key-card.tsx` should read/write `providers.anthropic.key` instead of the flat `anthropicApiKey`, but the UI label and layout stay the same.

4. **`App.tsx` `hasApiKey` check** — Currently checks `ANTHROPIC_API_KEY`. After SD-031, must check `providers[activeProvider]?.key` is truthy. This is a behavioral change and must be included in SD-031 (not deferred to SD-032) so the panel correctly gates Layer 2 for all three providers.

5. **`telemetry.ts` counter keys** — `analyze_invalid_key`, `analyze_rate_limit` are provider-neutral string names. No change needed. But `key_saved` and `key_rejected` should eventually be provider-tagged (future work).

6. **`rubric_v1.0.prompt.txt`** — This file exists in `extension/src/service-worker/` but is NOT used at runtime. The actual prompt is the `PROMPT_TEMPLATE` const inside `rubric-prompt.ts`. The `.txt` file appears to be a reference copy. No action needed, but Plan-Writer should note: SD-033 and SD-034 will add their variant content inside `rubric-prompt.ts`, not as new `.txt` files.
